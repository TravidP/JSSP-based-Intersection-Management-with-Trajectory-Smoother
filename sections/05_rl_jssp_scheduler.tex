\subsection{Graph State and Operation Nodes}
% CN-SUBSECTION: 说明调度器看到的图状态，以及operation node和typed edge的含义。
\begin{addedtwoblock}
At each high-level scheduling event, the scheduler receives the operation graph $\mathcal{G}$ defined in Section~\ref{sec:problem_formulation}. Each node $v_{i,k}\in\mathcal{V}_g$ represents one operation: vehicle $i$ traversing conflict zone $z_{i,k}$. The graph contains typed structural relations rather than a single homogeneous edge set. Route-precedence relations encode the predecessor and successor context of the same vehicle, while disjunctive relations connect operations competing for the same conflict zone.

This representation follows learning-based JSSP dispatching on disjunctive graphs, but adapts the node features and relations to autonomous intersection management. A relation-aware graph attention encoder is used because the scheduler must distinguish same-vehicle motion continuity from same-zone conflict competition \cite{velickovic2018graphattention,zhang2020learningdispatch,shaw2018selfattention,schlichtkrull2018rgcn,chen2021rgat}.
\end{addedtwoblock}

\subsection{Operation Feature Encoding}
% CN-SUBSECTION: 归纳每个operation node的输入特征，并说明连续、类别、二值/mask特征如何编码。
\begin{addedtwoblock}
For an operation node $v=v_{i,k}$, the input representation summarizes vehicle $i$ before it traverses conflict zone $z_{i,k}$. The encoder deliberately separates scalar state variables, binary scheduling flags, and categorical identifiers:
\begin{itemize}
  \item \textit{Scalar state features}: motion, timing, delay, and route-progress quantities, including current speed $u_i(t)$, distance to the target zone $d_{i,k}(t)$, nominal free-flow entry time $\hat{s}^{\mathrm{ff}}_{i,k}$, occupancy time $p_{i,k}$, stop-line waiting time $W_i(t)$, incremental delay $\Delta D_i(t)$, route completion ratio $\rho_{i,k}$, and remaining-operation count $n_i^{\mathrm{rem}}$.
  \item \textit{Binary scheduling flags}: the fixed or committed indicator and the current schedulability indicator.
  \item \textit{Categorical identifiers}: operation status $\sigma_v$, conflict-zone index $z_v$, and vehicle class $\tau_v$. The status $\sigma_v$ indicates whether the operation is unscheduled, executing/committed, or scheduled/finished.
\end{itemize}
The constant-speed model is used only as a nominal free-flow reference for estimating $\hat{s}^{\mathrm{ff}}_{i,k}$ and $\Delta D_i(t)$; it is not assumed to be the final vehicle-control behavior. The scalar and binary feature vectors are
\begin{equation}
\begin{aligned}
  c_v = [&u_i(t), d_{i,k}(t), \hat{s}^{\mathrm{ff}}_{i,k}, p_{i,k}, W_i(t), \Delta D_i(t), \\
  &\rho_{i,k}, n_i^{\mathrm{rem}}],
\end{aligned}
\label{eq:continuous_features}
\end{equation}
\begin{equation}
  b_v=[\mathbb{1}_{\mathrm{fixed}}(v),\mathbb{1}_{\mathrm{sched}}(v)].
  \label{eq:binary_features}
\end{equation}
The encoder first forms a structured concatenated feature vector:
\begin{equation}
\begin{aligned}
  \tilde{x}_v = [&\operatorname{Norm}(c_v)\Vert b_v\Vert E_{\mathrm{status}}(\sigma_v)\Vert E_{\mathrm{zone}}(z_v) \\
  &\Vert E_{\mathrm{type}}(\tau_v)],
\end{aligned}
\label{eq:feature_concatenation}
\end{equation}
Thus, $\tilde{x}_v$ does not repeat $c_v$: it concatenates normalized scalar states, explicit binary scheduling flags, and learned categorical embeddings. The schedulability indicator is intentionally present both in $b_v$ and as the hard actor mask in \eqref{eq:actor_mask}, so the encoder can condition on feasibility while invalid actions remain impossible.\featureencodingpdfcomment
\begin{equation}
  h_v^{(0)}=\phi_{\mathrm{enc}}(\tilde{x}_v),
  \label{eq:input_encoder}
\end{equation}
where $\phi_{\mathrm{enc}}$ is the shared input projection that maps the concatenated operation features into the initial node embedding used by the relation-aware attention layers.
\end{addedtwoblock}
\subsection{MDP and Rollout Dynamics}
% CN-SUBSECTION: 用MDP语言定义observation、action、transition、terminal和reward，使训练流程可复现。
\begin{addedtwoblock}
The MDP state at rollout step $\ell$ is the current partial schedule and its graph representation, denoted by $S_\ell=(\mathcal{G}_\ell,\mathcal{P}_\ell,\mathcal{F})$, where $\mathcal{P}_\ell$ is the partial operation order and $\mathcal{F}$ is the set of fixed or committed operations. The action set is
\begin{equation}
  \mathcal{A}_\ell=\{v\in\mathcal{V}_g^{\mathrm{rem}} \mid \operatorname{Pred}(v)\subseteq \mathcal{P}_\ell\cup\mathcal{F}\},
  \label{eq:schedulable_action_set}
\end{equation}
so an action selects exactly one schedulable operation. The transition appends the selected operation to $\mathcal{P}_\ell$, updates node status and masks, and exposes any newly schedulable successors. The episode terminates when $\mathcal{V}_g^{\mathrm{rem}}=\emptyset$.

The primary rollout reward is the negative weighted incremental delay
\begin{equation}
  R_\ell = - \sum_{i\in\mathcal{V}(t)} w_i\Delta D_i^{(\ell)},
  \label{eq:delay_reward}
\end{equation}
where $\Delta D_i^{(\ell)}$ is the additional delay induced by the partial schedule at rollout step $\ell$ relative to nominal free-flow travel, and $w_i$ is an optional vehicle-priority weight. Stop penalties and smoother-infeasibility penalties are treated as ablation terms rather than mandatory components of the base reward.
\end{addedtwoblock}

\subsection{Relation-Aware Attention Policy}
% CN-SUBSECTION: 定义typed-edge消息传递、全局图信息、masked actor和可选critic。
\begin{addedtwoblock}
Let $\mathcal{R}_{\mathrm{edge}}=\{\mathrm{pre},\mathrm{succ},\mathrm{disj}\}$ denote predecessor, successor, and disjunctive same-zone relation types. For layer $k$, each relation produces a multi-head message
\begin{equation}
  m_{v,r}^{(k)} = \big\Vert_{a=1}^{H} \sum_{u\in\mathcal{N}_r(v)} \alpha_{uv,r}^{(k,a)} W_{r}^{(k,a)}h_u^{(k-1)},
  \label{eq:relation_message}
\end{equation}
where $H$ is the number of attention heads, $\mathcal{N}_r(v)$ is the relation-specific neighborhood, and $\alpha_{uv,r}^{(k,a)}$ is the normalized attention score. The node state is updated by
\begin{equation}
\begin{aligned}
  h_v^{(k)}=\phi_k(&[h_v^{(k-1)}\Vert h_v^{(0)}\Vert m_{v,\mathrm{pre}}^{(k)}\Vert m_{v,\mathrm{succ}}^{(k)} \\
  &\Vert m_{v,\mathrm{disj}}^{(k)}\Vert m_{\mathrm{glob}}^{(k)}]),
\end{aligned}
  \label{eq:gat_update}
\end{equation}
with $K=3$ update layers in the initial design. The global message $m_{\mathrm{glob}}^{(k)}$ is obtained by pooling all current node embeddings and broadcasting the graph-level context back to each operation.

The actor head maps final node embeddings to operation logits $\eta_v$. The schedulability mask is enforced as
\begin{equation}
  \eta_v=-\infty \quad \mathrm{if}\quad v\notin\mathcal{A}_\ell,
  \label{eq:actor_mask}
\end{equation}
followed by a softmax over the remaining feasible actions. The policy may sample during training and use greedy or beam-style rollout during evaluation. An optional critic or rollout baseline can reduce gradient variance, following attention-based combinatorial optimization practice \cite{kool2019attention}.
\end{addedtwoblock}

\subsection{Offline Training and Online Deployment}
% CN-SUBSECTION: 区分离线训练和在线部署，强调在线阶段只做冻结策略rollout。
\begin{addedtwoblock}
Let $\pi_\theta$ denote the trained RL policy. During offline training, the parameters $\theta$ are optimized using sampled JSSP/AIM instances and simulation feedback. During online deployment, $\theta$ is frozen. For a scheduling event at time $t$, the scheduler initializes the partial schedule with fixed commitments, computes the schedulable set, and repeatedly queries $\pi_\theta$ to select the next operation. The rollout terminates only when all remaining operations of the current active vehicles have been scheduled.

After the rollout produces a complete operation order, a deterministic schedule generator assigns target entry times $s_{i,k}$ and occupancy intervals using the selected order, route precedence constraints, fixed commitments, and estimated processing times $p_{i,k}$. The resulting reservation set $\mathcal{S}$ is then sent to the low-level smoother. Thus, the online stage is policy inference plus schedule generation, not online RL training.
\end{addedtwoblock}

\begin{addedtwoblock}
Training should proceed in three stages: abstract JSSP/AIM graph training, SUMO closed-loop validation or offline fine-tuning, and smoother-integrated evaluation. The implementation should log delay reward, feasibility, mask validity, rollout length, and smoother infeasibility to distinguish graph-construction errors from policy-learning failures before deployment.
\end{addedtwoblock}
